**Key conclusions:**

- For lifecycle: write-once per phase (Model II) selected — Ingest/Verify/Optimize/Codegen phases with frozen-after-write semantics; Model I (immutable after ingest) is the mandatory fallback for compliance-only profiles
- Section 04 spec file expanded with three new subsections (§4–§6) including code examples and beginner-friendly explanations of alternatives

**Decisions made:**
- Component schema: flat typed catalog (Option A) with six core component types defined (`NodeKindTag`, `SlaConstraint`, `CorePin`, `ProfileTag`, `EdgeTransport`, `MemoryDomain`)
- Query semantics: declarative YAML pass manifest required for each pass; executor builds concurrency graph from declared `reads`/`writes` sets
- Lifecycle: phase-gated mutation (Model II) is the default; immutable-after-ingest (Model I) is mandatory for compliance-only profiles
- Option C (tagged union categories) is acceptable ONLY for the Introspection Schema exposed to tooling
